## 一、病根：一句话说清楚

系统里独立发生了至少 5 次同一件事：

| 加的能力 | 本该做的事 | 实际做的事 |
|---|---|---|
| LIVE-RULES.md（规则文档） | 给模型参考，让它自己判断怎么用 | 每次整份 7000 字硬塞进 prompt，不管相不相关 |
| VIN 解析 | 查不到时，让模型用通用知识兜底推理 | 只查一个美国车管数据库，查不到就死，禁止模型自己推理 |
| 库存匹配（`findInventoryMatches`） | 没精确命中时，让模型判断"大概率能做" | 没命中 = 直接进"查不到"分支 |
| 转人工（`needs_price_confirmation`） | 系统兜底判断该不该转 | 完全依赖模型自己记得打标，经常没打，客户被静默放鸽子 |
| Coach 纠错闭环 | 龙哥纠错 → 立刻影响下一次回复 | 纠错变成工单，堆积 72 个没人处理；`good-examples/` 生成了 82 条，从来没被读进过 prompt |

**共同点：每一处都是"能力=覆盖判断"，不是"能力=辅助判断"。加得越多，模型的判断权被拿走得越多，于是工具越堆越多、答案越来越像个只会说"查不到/等团队"的傻子。**

这不是靠"再修一处"能治好的，因为下一个新功能大概率还会犯同样的错——需要的是一条统一的架构原则 + 一套所有能力都必须接入的公共基础设施。

## 二、完整架构：四层

### Layer 1｜硬红线（唯一允许"代码强制覆盖模型"的地方）

**判断标准：只有真金白银的合规/资损风险，才配得上"不许模型有例外"。**

- 不能报出没有真实依据的具体价格/库存数字
- 不能说"已核实/已确认"没有真的核实过的事
- 不能暴露内部工具/OCR/审批流程
- 不能把客户自己的号码回给客户当联系方式
- 不能把私人转人工号码当偷懒默认动作甩出去

控制在 10 条以内，代码强制拦截（现有 Reply Filter 保留），**这是全系统唯一保留"模型没有商量余地"的地方**。除此之外，其余所有东西都要下放到 Layer 3 交给模型判断。

**⚠️ 给实现者的红线判断标准（必须先看这条，再动 Layer 3）：** "放开推理"和"禁止编造"不矛盾，但两者的界线必须清晰、可执行，不能靠模型自己拿捏：

- **允许模型推理的，是"技术事实"**——VIN 结构、WMI 前缀、发动机型号对照、车型规格——这些是全球公开、任何一台同型号车都一样的客观标准，不属于"编造"，属于专业知识。
- **禁止模型推理/猜测的，是"业务私有事实"**——我们仓库里具体这台货还有没有、具体报价多少钱、具体多久能到——这些只有 AsiaPower 内部系统/团队知道，客观世界里没有"标准答案"可推理，模型没有依据就必须说没有依据，不能"合理推测"一个数字。
- 判断方法：**这件事换成任何一家做同类生意的公司，答案是不是都一样？是 → 技术事实，可以推理；不是（跟我们自己的库存/定价/物流有关）→ 业务私有事实，必须走 Layer 1/2 的真实证据，没有就不能说。**

### Layer 2｜证据组装（取代"整份规则硬塞"和"查不到就死"）

回复生成前，系统尽力拼出**所有能拿到的真实证据**，不管完不完整、成不成功，全部作为"证据"交给模型，而不是替模型下结论：

- 库存匹配结果——包括**近似匹配**（现在没做：没精确命中就什么都不给，应该改成"没精确命中但同品类有类似型号"也作为证据带上）
- VIN 解析结果——**包括失败的**：现在失败了只传一个 `status: failed`，应该把**原始 VIN 字符串本身 + 一个便宜的确定性校验结果**一起传下去（VIN 第 9 位是数学可验证的校验码，代码几行就能算出"哪一位大概率抄错了"，不需要模型）。**这条同样适用于客户提供的其它原始信息**（型号描述、客户口误的年份等）——不是"库存/数据库没覆盖到"才算证据缺口，"客户给的原始数据本身可能就是错的"也要连同确定性提示一起作为证据交给模型，而不是只在系统查不到时才暴露原始输入。
- 情境相关的规则片段（把 LIVE-RULES.md 按现有章节结构打标签，只挑跟当前对话阶段相关的部分，而不是整份 7000 字）
- 情境相关的历史好例子（`good-examples/` 82 条现成的，接进来，按标签检索，不用全读）
- 这个客户自己的历史消息（现在已经有，保留）

**这一层的核心变化：从"给模型一堆指令、一个已经算好的结论"，变成"给模型一堆真实但可能不完整的证据"。**

### Layer 3｜推理（把判断权还给模型）

明确告诉模型：**Layer 2 给你的证据可能不全，如果有缺口，用你自己的专业知识去推理，就像客户直接把问题贴给你、没有任何工具时你会做的那样**——VIN 解析失败时自己看 VIN 结构猜、猜到疑似打错字提出来；库存没精确命中但同系列有类似型号时可以说"这个系列我们大概率能做"；只有连"自己想"都想不出来的时候，才走 Layer 4 转人工，而不是把转人工当成第一反应。

这一层不需要新增复杂机制，核心是**去掉现在 prompt 里"只准转达 Layer 2 算好的结论、禁止自己推理"这类限制**——本质是做减法，不是加法。

### Layer 4｜转人工与学习闭环（确定性触发 + 答案自动复用）

- **转人工判断不再依赖模型记得打标**，改成系统根据客观信号确定性判断（Layer 2+3 都没能给出真实依据、或触碰了 Layer 1 红线），代码强制执行，不给模型"忘记"的机会。
- **分级转人工**：常规的"要跟中国供应商核实"→ 现成的 Ghana 团队通道（`ghana-staff-handoff.mjs`，已存在只是没接上）；真正陌生、系统判断不了的 → 龙哥，带具体问题，不是模糊 ping。
- **人工给出的答案，自动变成 Layer 2 的新证据**——龙哥或 Ghana 团队回答一次，这个答案被结构化存下来、打上标签，下次遇到类似情况，Layer 2 自动把它当证据带上，不用龙哥再答第二次。这一条同时也是"规则纠错"的答案：龙哥纠正一句话，走的是同一条入口，不再需要走"人工改 LIVE-RULES.md 文件"这条重工程路径。
- **⚠️ 跨客户复用的边界**：Layer 4 回写的是"人工给出的事实/判断"，不是"人工给某个客户的承诺"——比如龙哥回答"这台 2013 MINI Countryman 的 N18B16A 发动机我们能做"，可以作为技术判断复用；但如果是"给客户A优惠到800美元"这种针对单个客户的价格让步，**不能**被下一个客户的对话检索到当成默认报价。回写证据库时要区分"可复用的通用事实"和"仅对当次这个客户有效的承诺"，实现时这是一个必须做的判断，不是可选项。

### 2026-07-21 纠偏：Step 0 落地时偏离了 Layer 3/4 原意，这里写清楚具体边界

Step 0（`priceConfirmationGate`/`routePriceConfirmationHandoff`）上线后，观察到"没有现成库存/报价证据 = 立刻拦截转人工"这个行为——这不是 Layer 4 的本意，Layer 4 一开始就写着"只有连自己想都想不出来的时候，才走 Layer 4 转人工，而不是把转人工当成第一反应"。真实原因定位到 `bridge.mjs` `runOpenClawReply()` 的 prompt 里这条指令：

> "If inventory_matches is empty or has no good match, do not invent a number — say you'll confirm the price with the team, and set needs_price_confirmation to true."

这条把"不能编造具体数字"（Layer 1 红线，正确）和"必须说等团队、必须标记转人工"（行为强制）捆在一起了，模型没有机会像 Layer 3 设想的那样——像 ChatGPT 拿到同一个问题时那样，先用自己的专业知识/近似证据判断能不能自己给出有用的回答，只有真的想不出来才退到"等团队确认"这句话，而这句话本身也**不代表要转人工**，只是诚实的正常回复。

**修正后的转人工边界（具体化 Layer 4 的"确定性触发"）：**

- **Layer 1 红线拦截** ≠ **转人工**。模型编造了具体数字/库存/交期，拦截后应该让模型重新生成一句诚实的"确认中"类回复直接发给客户，不需要打断任何人。
- **真正需要转人工（加纳同事）的时机**：客户已经明确表达成交意向（代码里 `buying_intent_confirmed` 字段已经能识别，`sales_coach/detectors.py` 的 `closing` 意图分类同样已存在），下一步需要收货地址/安排当地取货或物流交接——这是唯一一类"必须有人介入才能完成"的信息缺口，因为地址这种信息不在任何数据库里，AI 想不出来、查不到，只能靠人工去问客户、去协调物流。
- **真正需要转龙哥（CEO）的时机**：维持 Layer 4 原意——"真正陌生、系统判断不了的"，不是价格/库存本身没有现成证据。
- 价格/库存没有现成精确匹配时，prompt 需要明确允许模型走 Layer 3 的推理路径（近似匹配/同系列型号可做/历史报价区间等），而不是一刀切地被摁着说"等团队"。

### 2026-07-21 补充：5W2H 节奏推进也是 Layer 3 推理的一部分，不是防复读补丁

现状核实：`apsales-soft-angle.mjs` 的 `detectPossibleRepeat()` 把 5W2H（why/when/where/how/how_much）做成了一个反复读安全网——只有最近两句 AI 回复都命中固定"still confirming/waiting for team"话术且高度重复时才触发，今天生产实测触发次数为 **0**（`apsales_soft_angle_used`/`apsales_soft_angle_dry_run` 全天计数都是0）。这不是这套机制的目标设错了范围的问题，是它跟龙哥要的东西根本不是一回事：龙哥要的是"每段对话模型自己心里有数、知道该往哪个维度推进"，是持续性的推理判断，不是"卡壳时的补救动作"。

**修正方向**：
- `uncoveredClosingAngles(dealState)`（已存在，计算还没问过的5W2H维度）不再只在 `possible_repeat_detected` 为真时才喂给模型，改成**每一轮都作为背景信息**交给模型，让它像 Layer 3 其它推理一样自己判断"这一轮要不要自然带一句"。
- `possible_repeat_detected` 从"唯一触发条件"降级为"其中一个强提示信号"（复读风险高时更应该用），不再是开关。
- `uncoveredClosingAngles()` 目前不看对话所在阶段，只看字段是不是空——需要按 LIVE-RULES.md"三段式"的依赖顺序过滤：车辆+配件（第一段）没锁定之前，不该被提醒去问港口/付款/数量（第三段），否则会在不该问的时候被推着问，显得生硬。
- 保留"不是打勾清单、不强制每题必问"这条原则不变，只是把决策权从固定触发条件交还给模型的判断。

## 三、为什么这是"一套"方案，不是四个补丁

Layer 2 和 Layer 4 本质是**同一套基础设施**：一个"证据库"，写入端是人工纠错/转人工答案/coach 发现的好例子，读取端是每次生成回复前的检索。建成这一套之后：

- VIN 失败 → 自动变成证据，不再是死胡同（不用单独修 VIN）
- 库存近似匹配 → 自动变成证据（不用单独修库存匹配）
- Coach 那 72 个积压工单、82 条好例子 → 直接是这套证据库的冷启动素材（不用单独修 coach）
- 转人工可靠性 → 因为触发条件从"模型自觉"变成"系统判断"，自动解决（不用单独修转人工）
- **未来任何新功能**，只要遵守"给模型证据、不替它下结论、失败也要暴露原始信息"这条原则接入 Layer 2，就不会重犯这个病——这才是"根治"，不是"这次的坑填了，下次挖新坑"。

## 四、落地顺序（一套方案里的分期，不是新一轮排查）

**第 0 步（独立、立刻做、风险最低）——转人工可靠性修复**：把 `needs_price_confirmation`（或等价判断）从"模型自己记得打标才生效"改成系统根据客观信号强制判断——这一步不涉及推理放开、不改变模型行为逻辑，纯粹是可靠性修复，跟 Layer 2/3 无依赖关系，**应该最先上线**，因为它现在是真实在漏客户（客户被承诺"团队会回复"，但因为标志没打上，后台没人收到通知）。不用等 Layer 2/3 做完。

1. **Layer 1 收窄**——先把现有几十条零散规则里，真正的硬红线挑出来，改成代码强制；其余全部降级为 Layer 2 的"参考证据"。
2. **Layer 2 证据组装器**——新建一个统一的"证据组装"步骤，取代现在分散在 `bridge.mjs` 里的各处硬编码逻辑（VIN 结果只传状态、库存没命中就不传、规则整份贴），改成"有什么真实信息都往上带，包括不完整/失败的"。
3. **Layer 3 松绑 prompt**——去掉"只能转达算好的结论""禁止自己推理"这类限制性指令，同时写入"技术事实 vs 业务私有事实"这条边界（见上文红线判断标准）。
4. **Layer 4 剩余部分：分级转人工 + 证据回写**——接入 Ghana 团队分级通道；人工答案自动写回 Layer 2 的证据库，注意上面"跨客户复用边界"这条。

第 0 步之外的四步是一套东西的四个安装步骤，不是四个独立项目——顺序上必须先有 Layer 2（证据组装）这个地基，Layer 3/4 才有地方接。

## 五、上线安全措施（客户端是真实线上渠道，不能直接切）

- **先用现成的 sandbox 模式验证**：代码库里已经有 `WHATSAPP_AUTONOMY_MODE=sandbox`（CEO 白名单测试模式，`docs/tasks/apsales-decision-001/analysis.md` 有说明），新逻辑先在这个模式下跑，不直接上生产。
- **用真实历史数据回放做回归测试**，不要只凭感觉判断"看起来对了"：`data/evidence/whatsapp/turns.ndjson` 里已经有几百条真实对话，把新旧两版 pipeline 跑同一批历史客户消息，对比输出，重点看第六节列的具体案例。
- **改动 `bridge.mjs` 前先确认当前生产版本可以快速回滚**（git revert 即可恢复），上线后第一个 24 小时密切盯 Telegram 转人工通知量和客户投诉，异常就回滚，不要现场调参数。
- **不要重新引入"让模型自己实时调用工具查询"这种模式**——之前不做是因为库存有 500+ 条、800KB，实时查太慢太不可靠，这个判断依然成立。本方案的"证据组装"始终是服务器端提前查好、打包好再交给模型，不是开放实时工具调用，实现时不要为了"看起来更 agentic"就换成实时工具调用。
- **本次改动必须以真实、可核实的 Git 产物交付**：真实分支 + 真实 commit（`git log` 能看到）+ 真实测试命令与输出（不是复述数字）。完成报告里引用的每一个文件路径，交付前必须自己先用 `ls`/`git show` 验证一遍存在且被正确引用，而不是等审核方发现文件不存在。

## 七、诚实的局限性（不是"完美方案"，是"当前信息下最优判断"）

- **这套方案用"推理"替代"死板规则"，天然会牺牲一部分确定性**——模型偶尔可能推理错（比如校验位提示模糊、有两种可能纠正方式时选错一种），这跟死规则"每次都可预测、但经常显得笨"是真实的权衡取舍，不是这次改完就能连带消除的风险；靠的是回归测试 + 持续把人工纠正喂回证据库来收敛，不是一次上线就完美。
- **证据库/检索层是全新基础设施，工作量比单独修 VIN 或修转人工大得多**——如果时间/资源有限，第 0 步（转人工可靠性）+ Layer 1（红线收窄）可以独立先做，价值也是实打实的，不必等一整套 Layer 2 证据库建完才上线任何东西。
- **VIN 校验位纠错本身不保证 100% 猜对**——只是把"没有任何线索瞎猜"变成"有数学依据的线索去推理"，跟 ChatGPT 那次答对一样，是大概率答对，不是保证答对；客户提供的信息如果错得比较离谱（不止一位错），这条也救不回来，遇到这种情况仍然应该走 Layer 4 转人工，而不是死磕推理。
